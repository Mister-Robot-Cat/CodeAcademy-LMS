# CodeAcademy LMS — Мастер-документ для работы с AI-агентом (Antigravity / Gemini 3.1 Pro)

Этот файл — твой постоянный контекст для агента. Вставляй его **целиком** в начале каждой новой сессии/чата с агентом, прежде чем давать конкретную задачу. Так агент не будет "терять память" о проекте между сессиями.

---

## 0. Как этим пользоваться (коротко)

1. В начале новой сессии с агентом — вставь блок **"1. КОНТЕКСТ ПРОЕКТА"** целиком.
2. Затем вставь **ОДНУ** фазу из **"3. РОАДМАП"** (не все сразу!).
3. В конце добавь блок **"4. ПРАВИЛА РАБОТЫ ДЛЯ АГЕНТА"**.
4. Дай агенту закончить фазу, проверь руками, закоммить, и только потом переходи к следующей фазе новой сессией.

Никогда не проси "сделай весь проект целиком" одним промптом — агент физически не удержит такой объём контекста без ошибок, дублирования кода и поломки уже готовых модулей.

Ты — senior full-stack разработчик, работающий над проектом CodeAcademy LMS.
Ты работаешь в IDE с прямым доступом к файловой системе и терминалу (PowerShell).

CodeAcademy LMS — система управления учебным процессом (LMS) для учебных заведений.
Целевые роли: SuperAdmin, Admin, Teacher, Student, Parent, Curator, Accountant.
Полное видение продукта описано в разделе "Полный бэклог" ниже — но сейчас мы

- CQRS через MediatR
- PostgreSQL через Entity Framework Core (Fluent API, миграции)
- JWT Bearer аутентификация
- Путь: C:\Users\user\Documents\GitHub\CodeAcademy LMS\src\backend
- ВАЖНО: запускать строго командой `dotnet run --launch-profile https`
  (иначе поднимется http:5232 вместо https:7109, а фронт ждёт https:7109)
- Пароль от локальной БД PostgreSQL: 123123

Уже реализовано на бэкенде:
- /api/auth/register, /api/auth/login (JWT)
- CRUD + пагинация: /api/student, /api/teacher, /api/group/semesters
- /api/group: создание групп, зачисление студентов (M2M через Enrollment),
  назначение учителей на группу с предметом (M2M через GroupTeacher)
- /api/admin/stats — статистика для дашборда
- Списки возвращаются в формате PaginatedList (обёрнуты в свойство items)

Frontend (в активной разработке, лежит в src/frontend):
- React 19, Next.js 16 (App Router), TypeScript, TailwindCSS v4
- Дизайн: тёмная тема, glassmorphism (полупрозрачные карточки, backdrop-blur,
  микро-анимации), премиальный современный стиль

- Путь: C:\Users\user\Documents\GitHub\CodeAcademy LMS\src\frontend

Уже реализовано на фронте:
- Страницы логина/регистрации
- Layout админ-панели с боковым меню
- Дашборд со статистикой
- Полностью готова страница /admin/groups с кастомными компонентами:
  ActionModal (модалки), SearchableCombobox (поиск в select),
  FloatingLabelInput (инпуты с плавающими лейблами)

### Технические нюансы, которые нужно учитывать
- JWT-токен (accessToken) хранится во фронтенде в localStorage.
- После любого пересоздания БД старые токены становятся невалидными (401) —
  это нормально, требуется просто перелогиниться, это НЕ баг в коде.
- Все списковые ответы API — PaginatedList с полем items, фронт это уже парсит,
  сохраняй этот же паттерн для новых эндпоинтов.
- Следуй уже заданному стилю кода и структуре папок, не изобретай новую
  архитектуру для новых модулей — расширяй существующую.

## 3. РОАДМАП ПО ФАЗАМ (давать агенту по одной фазе за сессию)

### Фаза 1 — Электронный журнал + Посещаемость (ядро продукта)
Backend: сущности Lesson, Attendance; CRUD уроков внутри группы/предмета;
проставление посещаемости по студентам группы; авто-расчёт % посещаемости.
Frontend: страница журнала для учителя (список уроков группы, форма посещаемости),
страница просмотра посещаемости для админа.

### Фаза 2 — Оценки и успеваемость
Backend: сущность Grade (типы: quiz/exam/assignment/lab/...), привязка к
Lesson/Subject/Student; расчёт GPA и рейтинга группы.
Frontend: выставление оценок учителем, страница успеваемости для админа/студента.

### Фаза 3 — Портал студента
Backend: эндпоинты "мои данные" (расписание, оценки, посещаемость) с учётом роли.
Frontend: отдельный layout для роли Student, дашборд с графиками (успеваемость,
посещаемость по месяцам).

### Фаза 4 — Портал родителя
Backend: связка Parent–Student, эндпоинты для родителя (read-only доступ к данным ребёнка).
Frontend: layout для роли Parent.

### Фаза 5 — Задания (LMS)
Backend: сущность Assignment (дедлайн, файлы), Submission (сдача студентом).
Frontend: создание заданий учителем, сдача заданий студентом, страница оценивания.

### Фаза 6 — Уведомления
Backend: сущность Notification + сохранение в БД. WebSocket или SignalR для real-time.
Frontend: колокольчик уведомлений, страница всех уведомлений.

### Фаза 7 — Календарь
Backend: агрегирующий эндпоинт (уроки + экзамены + события) по датам.
Frontend: компонент календаря (месяц/неделя) с фильтрами.

### Фаза 8 — Финансы (упрощённая версия для начала)
Backend: сущность Payment/Invoice, статус оплаты по студенту.
Frontend: страница финансов для админа/бухгалтера, статус оплаты в профиле студента.

### Фаза 9 — Отчёты и экспорт
Backend: генерация отчётов (посещаемость, успеваемость, финансы) в PDF/Excel.
Frontend: кнопки экспорта на соответствующих страницах.

### Фаза 10+ (bonus, делать в последнюю очередь и по желанию)
Мессенджер, онлайн-экзамены, сертификаты с QR, ИИ-аналитика, 2FA, мобильное
приложение, распознавание лиц, геймификация.

> Совет: не переходи к следующей фазе, пока текущая не собирается без ошибок
> и ты не проверил её руками (создание урока → выставление оценки → просмотр
> статистики и т.д. — весь пользовательский путь).

## 5. АВТОНОМНЫЙ РЕЖИМ (для scheduled-запусков)

Используется, когда агент запускается по расписанию БЕЗ тебя рядом. В этом
режиме PROGRESS.md — единственная память агента между запусками, поэтому
правила ниже обязательны и не подлежат сокращению.

```
Ты работаешь в АВТОНОМНОМ режиме без человека рядом. Действуй консервативно:
лучше сделать меньше, но правильно, чем много и сломать проект.

ПЕРЕД началом работы — обязательно:
1. Прочитай файл PROGRESS.md целиком.
2. Определи "Текущую активную фазу" — работай ТОЛЬКО в её рамках.
   Если статус фазы DONE_AWAITING_REVIEW или MERGED — ничего не делай,
   останови работу и запиши в журнал прогонов: "Ожидание ревью человеком,
   новая фаза не активирована — работа не начата".
3. Проверь секцию "Уже реализовано и стабильно" — эти модули НЕ трогать.

ВО ВРЕМЯ работы — жёсткие ограничители:
4. Работай только в git-ветке, указанной в PROGRESS.md для текущей фазы.
   Никогда не коммить и не мёржи в main/master напрямую.
   Если такой ветки нет — создай её от актуального main.
5. Если для работы нужна миграция БД — СОЗДАЙ файл миграции, но НЕ применяй
   её (`dotnet ef database update` НЕ запускать). Примени применение миграции
   как отдельный явный TODO для человека.
6. Перед коммитом обязательно: проект должен собираться (build PASS) и
   существующие тесты должны проходить. Если build/тесты падают и ты не
   можешь починить за разумное время — НЕ коммить, откати незавершённые
   изменения, зафиксируй проблему в PROGRESS.md со статусом BLOCKED.
7. Не начинай следующую фазу, даже если текущую закончил раньше срока.
   Одна фаза = один прогон (или несколько прогонов, если фаза большая —
   тогда статус остаётся IN_PROGRESS до полного завершения).
8. Не удаляй и не переписывай уже реализованные модули из секции
   "Уже реализовано и стабильно" без явной пометки в PROGRESS.md о том,
   что человек разрешил рефакторинг.

ПОСЛЕ работы — обязательно обнови PROGRESS.md:
9. Добавь новую запись в "Журнал прогонов" СВЕРХУ (не удаляя старые записи):
   дата, что сделано, какие файлы, статус миграций, build PASS/FAIL,
   что осталось, что заблокировано.
10. Обнови статус фазы в таблице "Карта фаз":
    - IN_PROGRESS — если работа не закончена, есть что доделывать
    - BLOCKED — если не смог продолжить, нужна помощь человека
    - DONE_AWAITING_REVIEW — если фаза полностью реализована, build PASS,
      ветка готова к ревью и мёржу человеком (сам НЕ мёржи!)
11. Если статус DONE_AWAITING_REVIEW — останови автономную работу над
    следующими фазами, пока человек не смёржит ветку и не поменяет статус
    вручную.

Никогда не додумывай задачи, которых нет в текущей активной фазе, даже
если "напрашивается" логичное продолжение. Лучше остановиться и оставить
явный TODO в PROGRESS.md, чем самовольно расширять scope.
```

### Готовый промпт для scheduled-задачи (короткий, вставляй в настройки расписания)

```
Открой и прочитай PROGRESS.md в корне проекта — это твоя единственная
память между запусками. Открой CodeAcademy_LMS_AI_Agent_Brief.md — это
контекст проекта и правила работы. Следуй разделу "5. АВТОНОМНЫЙ РЕЖИМ"
из брифа строго и без исключений. Работай только над фазой, указанной как
активная в PROGRESS.md, только в отдельной git-ветке, без автоприменения
миграций БД и без мёржа в main. В конце обнови PROGRESS.md.
```
